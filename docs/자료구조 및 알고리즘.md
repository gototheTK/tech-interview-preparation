
## Q) 배열(Array)과 연결 리스트(LinkedList)의 메모리 할당 구조의 차이를 설명하고, 특정 데이터 탐색과 중간 삽입/삭제 시 발생하는 시간 복잡도(O)를 비교해주세요.


배열은 메모리 공간에 데이터가 물리적으로 연속해서 할당되는 구조이고, 연결 리스트는 데이터(노드)들이 메모리 곳곳에 흩어져 있으면서 다음 데이터의 주소(참조)를 기억하며 논리적으로만 연결된 구조입니다.
탐색의 경우, 배열은 물리적 순서(인덱스)를 알면 메모리 주소를 바로 계산할 수 있어서 접근 속도는 O(1)의 시간 복잡도를 가지지만, 연결 리스트는 임의의 접근이 불가능하여 맨 앞의 헤드부터 순차적으로 접근하여야 합니다. 그래서 연결 리스트의 탐색 시간 복잡도는 O(N)입니다.
중간 데이터 삽입/삭제의 경우에도 두 자료구조 모두 O(N)의 시간 복잡도를 가집니다. 배열은 데이터를 지우거나 넣은 후 나머지 데이터를 한 칸씩 당기거나 밀어내야 해서 O(N)이 걸리고, 연결 리스트는 포인터만 바꾸면 되지만 그 위치까지 순차적으로 탐색해서 찾아가는 과정이 선행되어야 하므로 결국 O(N)이 소요됩니다.


## Q) HashMap이 데이터를 빠르게 탐색하는 원리와, 서로 다른 키(Key)가 같은 해시 값을 가질 때 발새하는 ‘해시 충돌(Hash Collision)’을 자바(Java)에서 어떻게 해결하는지 설명해주세요.


HashMap은 '해시 함수'를 통해 입력값을 받으면 고정된 길이의 해시값으로 변환하고, 이를 배열(버킷)의 인덱스로 활용합니다. 그래서 접근 속도는 O(1) 시간 복잡도로 상당히 빠릅니다. 다만, 해시 함수로 만들어진 인덱스가 같아 충돌하는 경우가 있는데, 이를 '해시 충돌(Hash Collision)'이라고 합니다. 자바에서는 이를 해결하기 위해 '체이닝' 기법을 사용하는데, 원리는 버킷에서 충돌한 값들을 연결 리스트로 길게 매달아 저장해 이를 해결합니다. 그리고 한 버킷에 저장한 값의 개수가 8개 이상이 되면, 탐색 성능을 유지하기 위해 연결 리스트를 레드-블랙 트리(Red-Black Tree)로 변환하여 저장합니다.


## Q) 스택(Stack)과 큐(Queue)의 데이터 입출력 방식을 비유를 들어 설명하고, 웹 브라우저 ‘뒤로가기/앞으로 가기’ 기능이 이 중 어떤 자료구조를 사용하는지 그 이유를 설명해 주세요.


스택은 LIFO(Last In, First Out) 구조로, 컨테이너에서 가장 마지막으로 적재한 화물을 가장 먼저 꺼내는 것과 같습니다. 큐는 FIFO(First In, First Out) 구조로, 맛집의 대기줄처럼 가장 먼저 온 사람이 가장 먼저 입장하는 것과 같습니다.
웹 브라우저의 '뒤로 가기'와 '앞으로 가기' 기능은 과거의 이전 페이지와 다음 페이지들을 각각의 전용 스택(Back Stack, Forward Stack)에 넣어 Pop/Push 하며 이동시켜 주는 기능입니다.


## Q) 운영체제(OS)의 프로세스 스케줄링 대기열(Ready Queue)에 스택 대신 큐를 사용하는 이유와, 만약 스택을 사용했을 때 발생하는 치명적인 문제점을 설명해주세요.


OS의 프로세스 스케줄링은 먼저 온 프로세스의 요청을 처리하는 '큐'라는 자료구조를 사용하여야 합니다. 만약 큐를 사용하지 않고 스택을 사용하게 된다면 공정성에 큰 문제가 생기고, 새로운 요청이 끊임없이 오게 된다면 처음에 도착한 프로세스의 요청은 영원히 처리되지 못하는 **'기아 상태(Starvation)'**가 되어버리고 말 것입니다.


## Q) 트리(Tree)와 일반 그래프(Graph)의 결정적인 차이점은 무엇인가요?


트리는 부모와 자식 관계로 이루어진 계층 구조의 단방향 그래프입니다. 일반적인 그래프와의 차이점은, 트리는 루트 노드에서 시작하여 뻗어나가며, 두 노드 간의 경로가 하나뿐인 비순환 그래프라는 점입니다.


## Q) DFS와 BFS의 동작 원리와 자료구조의 차이, 그리고 각각 어떤 실전 상황에서 사용하는 것이 유리한가요?


DFS는 한 경로만 깊게 탐색하다가 막히면 최근 갈래로 되돌아와 다시 탐색하는 알고리즘으로, 내부적으로 LIFO 구조의 스택을 사용합니다. 그에 반해 BFS는 인접한 노드들을 발견한 순서대로 처리하여 퍼져나가는 알고리즘으로, 내부적으로는 FIFO 구조의 큐를 사용합니다. 


최단 경로 문제에서는 **BFS**가 적절합니다. 가장 먼저 도착한 경로가 최단 경로임을 보장하기 때문입니다. 만약 DFS를 사용한다면 이리저리 헤매다가 도착할 것이고, 최단 경로임을 보장하지 못할 것입니다. 


모든 경우의 수 탐색 문제에서는 **DFS**가 적절합니다. 모든 경우의 수를 보면서 필요 없는 경우의 수는 가지치기하여 메모리를 효율적으로 사용할 것입니다. 만약 BFS를 사용하면 경우의 수가 기하급수적으로 커지는 바둑이나 체스에서 필요 없는 경우의 수를 가지치기를 하지 못하고 수많은 갈래를 다 담아야 하므로, 결국 메모리가 뻗어버리는 OOM이 발생할 것입니다.
