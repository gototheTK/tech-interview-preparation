
## Q) 분산 서버 환경에서 Session 대신 JWT(JSON Web Token)을 사용할 때 얻을 수 있는 아키텍처 관점의 가장 큰 이점은 무엇인가요?


서버가 인증을 위해 외부 저장소(DB, Redis 등)를 조회할 필요 없이, CPU 연산(서명 검증)만으로 인증이 끝나 확장성이 뛰어납니다. 즉, JWT의 무상태성(Stateless) 덕분에 별도의 세션 저장소를 공유할 필요 없이 서버 증설(Scale-out)이 매우 자유롭습니다.


오답) JWT는 헤더, 페이로드, 서명 등 메타데이터가 포함되어 있어 일반적으로 세션 ID(해시값)보다 길이가 훨씬 길어 네트워크 트래픽이 늘어납니다.


## Q) READ UNCOMMITTED 격리 수준에서 발생하는 문제로, 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽었다가 해당 트랜잭션이 롤백될 경우 심각한 정합성 오류를 낳는 이상 현상은 무엇인가요?


더티 리드(Dirty Read)입니다. 커밋되지 않은 찌꺼기(Dirty) 데이터를 읽어와서 발생하는 문제입니다.


오답) 논 리피터블 리드(Non-Repeatable Read)은 기존 데이터가 변경/커밋되어 다시 읽었을 때 값이 달라지는 현상입니다. READ COMMITED격리 수준까지 일어나는 현상입니다.


## Q) 트랜잭션 격리 수준의 문제를 방어하기 위해 애플리케이션 코드 레벨에서 버전(Version) 정보 등을 활용하여 충돌을 막는 메커니즘은 다음 중 어느 것에 속하나요?


낙관적 락입니다. 낙관적 락은 실제 DB 락을 걸지 않고, 데이터 수정 시 버전 번호를 대조하여 충돌을 감지(버전업)하는 애플리케이션 레벨의 제어 방식입니다. 보통 충돌이 자주 발생하지 않을 것이라고 낙관적으로 예상되는 곳에 씁니다. DB에 실제 락을 걸지 않아 속도가 매우 빠르기 때문입니다. 하지만, 선착순 100명 쿠폰 발급 처럼 충돌이 미친 듯이 발생하는 곳에 쓰면, 예외(Exception)가 너무 많이 터지고 재시도 로직이 꼬여서 오히려 성능이 박살납니다. 그럴 때는 진짜 자물쇠를 거는 비관적 락(Pessimistic Lock)이나 Redis 분산락을 써야 합니다.


## Q) DB 실행 계획(EXPLAIN)을 뜯어봤더니, `type: ALL`지표와 `Extra: Using filesort` 지표가 동시에 떠버렸습니다.


### 1. 이 두 지표가 합쳐졌을 때, 서버 내부에서는 물리적으로 어떤 최악의 행동을 하고 있다는 뜻인가요?


### 2. 백엔드 개발자인 님이 이 쿼리를 살리기 위해 가장 먼저 취해야할 조치는 무엇인가요?


type:ALL은 쿼리를 처리할 때 테이블의 모든 튜플을 조회하여 처리하였다는 의미이고, Extra:Using filesort는 데이터를 정렬할 때 테이블의 모든 튜플들을 조회하여 정렬하였다는 의미입니다. 모두 적절한 인덱스가 없었기때문에 발생하는 문제로, 컬럼에 적절한 인덱스를 생성해준다면 해결됩니다.


## Q) 클라이언트와 서버가 TCP 3-Way Handshake를 성공적으로 마치고 연결되어 데이터를 주고 받고 있습니다. 그런데 갑자기 클라이언트 측 컴퓨터의 전원 코드가 발에 걸려 툭 뽑혀버렸습니다. 


## 이때, 서버는 클라이언트가 갑자기 죽엇다는 사실을 즉시 알아챌 수 있을까요? 만약 모른다면 서버쪽에 할당된 연결 자원(소켓, 메모리 등)은 어떻게 되며, 이를 방지하기 위해 네트워크에는 어떤 메커니즘이 존재할까요?


서버는 죽은 연결인 반쪽 연결((Half-Open, Dead Peer)상태의 소켓이 메모리를 계속 점유하는 것을 방지하기위해서, TCP Keep-Alive라는 타이머 메커니즘을 사용합니다. 원리는 일정 시간 동안 통신이 없으면, 서버는 빈 패킷(ACK/Probe)를 보내어서 찔러봅니다. 그래서 ACK 응답이 없으면 Timout이라고 판단하여, 소켓을 끊어버리고 자원을 회수합니다.


## Q) 여러 유저가 동시에 한정판 상품을 결제할 때 발생할 수 있는 동시성 문제와, 이를 애플리케이션 레벨에서 해결하는 낙관적 락(Optimistic Lock)의 원리에 대해 설명해 주세요.


아무런 처리를 하지 않으면 여러 트랜잭션이 마구잡이로 갱신하여서 재고가 마이너스가 되는 불상사가 발생합니다. 흔히 갱신 손실(Lost Update)이라고 불립니다. 이를 해결하기 위한 한 가지 방법으로 애플리케이션 수준에서 동시성을 제어하는 낙관적 락이라는 기법이 존재합니다. 이 낙관적 락의 원리는 먼저 테이블에 버전(@Version)을 추적하기 위한 컬럼을 지정하여, 조회 후 갱신 시 버전을 대조하여 동일하면 갱신하고, 동일하지 않으면 처리를 튕겨냅니다.


## Q) 스프링 AOP를 활용한 @Transactional과 관련하여, 다음 중 트랜잭션이 ‘적용되지 않는’ 치명적인 상황은 언제인가요?


같은 클래스 내부에 있는 @Transactional이 없는 A 메서드가, @Transactional이 있는 B 메서드를 내부적으로 호출할 때(Self-Invocation) 입니다. 내부 호출 시에는 스프링 프록시 객체를 거치지 않고 실제 객체(this)의 메서드를 직접 실행하기 때문에 기능(트랜잭션)이 발동하지 않습니다.
