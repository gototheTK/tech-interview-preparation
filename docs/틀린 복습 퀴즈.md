
## Q) 분산 서버 환경에서 Session 대신 JWT(JSON Web Token)을 사용할 때 얻을 수 있는 아키텍처 관점의 가장 큰 이점은 무엇인가요?


서버가 인증을 위해 외부 저장소(DB, Redis 등)를 조회할 필요 없이, CPU 연산(서명 검증)만으로 인증이 끝나 확장성이 뛰어납니다. 즉, JWT의 무상태성(Stateless) 덕분에 별도의 세션 저장소를 공유할 필요 없이 서버 증설(Scale-out)이 매우 자유롭습니다.


오답) JWT는 헤더, 페이로드, 서명 등 메타데이터가 포함되어 있어 일반적으로 세션 ID(해시값)보다 길이가 훨씬 길어 네트워크 트래픽이 늘어납니다.


## Q) READ UNCOMMITTED 격리 수준에서 발생하는 문제로, 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽었다가 해당 트랜잭션이 롤백될 경우 심각한 정합성 오류를 낳는 이상 현상은 무엇인가요?


더티 리드(Dirty Read)입니다. 커밋되지 않은 찌꺼기(Dirty) 데이터를 읽어와서 발생하는 문제입니다.


오답) 논 리피터블 리드(Non-Repeatable Read)은 기존 데이터가 변경/커밋되어 다시 읽었을 때 값이 달라지는 현상입니다. READ COMMITED격리 수준까지 일어나는 현상입니다.


## Q) 트랜잭션 격리 수준의 문제를 방어하기 위해 애플리케이션 코드 레벨에서 버전(Version) 정보 등을 활용하여 충돌을 막는 메커니즘은 다음 중 어느 것에 속하나요?


낙관적 락입니다. 낙관적 락은 실제 DB 락을 걸지 않고, 데이터 수정 시 버전 번호를 대조하여 충돌을 감지(버전업)하는 애플리케이션 레벨의 제어 방식입니다. 보통 충돌이 자주 발생하지 않을 것이라고 낙관적으로 예상되는 곳에 씁니다. DB에 실제 락을 걸지 않아 속도가 매우 빠르기 때문입니다. 하지만, 선착순 100명 쿠폰 발급 처럼 충돌이 미친 듯이 발생하는 곳에 쓰면, 예외(Exception)가 너무 많이 터지고 재시도 로직이 꼬여서 오히려 성능이 박살납니다. 그럴 때는 진짜 자물쇠를 거는 비관적 락(Pessimistic Lock)이나 Redis 분산락을 써야 합니다.
