
## 인터페이스와 추상 클래스의 본질적인 차이는 무엇인가요?


	> 추상 클래스는 객체의 종류와 공통된 상태를 정의합니다. 자바에서는 다중 상속이 불가능합니다. 그에 반해 인터페이스는 객체의 역할을 정희하고, 상태를 가질 수 없으며 다중 상속이 가능합니다.


## 구현체가 1개일 때도 무조건 인터페이스를 만드는 것이 좋은 설계인가요?


	> 아닙니다. 과거 스프링에서는 AOP(@Transactional 등)를 적용하기 위해 JDK 동적 프록시를 사용했기 때문에 강제로 인터페이스가 필요했습니다. 하지만 Spring Boot 2.0부터 CGLIB(Code Generation Library)가 기본값으로 채택되면서, 클래스를 상속받아 프록시를 생성할 수 있게 되었습니다. 따라서 무분별한 인터페이스의 생성은 파일 탐색 비용만 높이므로, 확장이 불필요 하다면 클래스 단독으로 사용하는 것이 실용적입니다.


## Spring 프록시 패턴과 템플릿 콜백 패턴(JdbcTemplate)의 결정적 차이는 무엇인가요?


	> **프록시 패턴(@Transactional)**은 타겟 코드를 수정하지 않고 겉에서 몰래 감싸(Wrapping) 부가 기능(트랜잭션 앞뒤 처리)을 추가합니다. 타겟 객체는 자신이 감싸진 줄도 모르며 수동적으로 동작합니다. 그에비해, **템플릿 콜백 패턴(JdbcTemplate)**은 변하지 않는 전체 흐름(DB 연결/해제 뼈대)을 미리 만들어 두고, 개발자가 능동적으로 템플릿을 호출하며 바뀔 부분(SQL 쿼리)만 파라미터(콜랙)로 던져 빈칸을 채우는 방식입니다.


## 생성자 주입(Constructor Injection)을 권장하는 3가지 이유는 무엇인가요?


	> 1. 불변성 보장: final 키워드를 사용하여 객체 생성 시점에 의존성을 고정할 수 있습니다.  
	>   
	> 2. 순환 참조 방지: A가 B를, B가 A를 참조할 경우, 컴파일 후 애플리케이션 기동(초기화) 시점에 스프링부트가 에러를 뿜어 시한폭탄을 막아줍니다.  
	>   
	> 3. 테스트 용이성: 스프링 컨테이너 없이도 순수 자바 코드로 가짜 객체(Mock)를 주힙해 단위 테스트를 하기 쉽습니다.
