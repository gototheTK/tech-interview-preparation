
## Union Find (Java Code)


```java
public class UnionFind {

	public int find (int[] arr, int i) {
			
			if (arr[i] == i) return i;
			
			return arr[i] = find(arr, arr[i]);
			
	}
	
	public void union(int[] arr, int a, int b) {
			
			int rootA = find(arr, a);
			int rootB = find(arr, b);
			
			if (rootA!=rootB) {
					
					if (rootA < rootB) {
							arr[rootB] = rootA;
					}else {
							arr[rootA] = rootB;
					}
					
			}
			
	}
	
}
```


## DFS 고려 사항

1. **명확한 탈출 조건(Base Condition)**
	- 함수 종료시키는 조건이 최상단에 꼭 있어야한다.
2. **철저한 방문 처리(Visited Check)**
	- 트리 구조가 아니라면 그래프에 무조건 순환이 존재할 수 있다고 의심해야한다.
3. **왔던 길 되돌리기(Backtracking)**
	- 모든 노드를 한 번씩 찍고 오는것이 아니라 가능한 모든 경로의 수를 구해야 한다면, visited[node] = false;와 같이 도장 및 상태복구 처리는 필수적이다.
4. **자바의 콜 스택(Call Stack) 한계 의식하기**
	- 자바 환경에서 재귀의 깊이가 보통 1만번을 넘어간다 싶으면, Stack이나 BFS를 쓰는걸 고려해야한다. 가령, 노드가 10만개인데 1자형으로 늘러져있다면, 실행 시 StackOverflowError를 발생시킬것이다.

## BFS(너비 우선 탐색) 실전 4대 원칙

1. **최단 거리는 무조건 BFS (동신원 탐색)**
	- DFS는 한 우물만 파는 ‘직진형’이라면, BFS는 잔잔한 호수에 돌을 던졌을 때 퍼지는 ‘물결’이라고 비유할 수 있다. 즉, 미로찾기에서 1칸을 움직이면 다음 칸들로 공평하게 넓혀가기 때문에, 목표 지점에 가장 먼저 도착하는 물결이 최단거리이다.
2. **인터페이스는 Queue, 인스턴스는 ArrayDeque**
	- 선입선출 자료구조인큐를 이용하되, 자바에서는 인스턴스로 ArrayDeque를 사용하도록 한다.
3. **방문 도장은 큐에 ‘넣을 때’ 찍는다.**
	- 만약 방문 도장을 ‘꺼낼 때’ 찍는 다면, 대기열을 기다리는 동안 다른 노드들이 똑같은 이웃을 큐에 중복해서 넣는 대참사가 발생한다. 그러니 특별한 상황이 아니면, 되도록이면 ‘넣는 순간’에 방문 도장을 찍도록한다.
4. **극한의 객체 다이어트**
	- 특수한 상황이 아니라면,  되도록이면 원시타입 내지는 원시타입배열을 쓰도록한다.

## 배열과 Wrapper 클래스의 주의사항

- 객체는 배꼽보다 배가 더 큰 객체 헤더라는 꼬리표를 가지고 다닌다.(64비트 JVM 압축 포인터 기준, 약 12~16바이트)
	- Integer 객체 1개: 객체 헤더(약 12~16바이트) + 실제 int 데이터(4바이트) = 총 16~18바이트
	- int[2] 배열 객체 1개: 객체헤더(약 12~16바이트) + 배열 길이 정보(4바이트)  + 실제 int 데이터 2개(8바이트) = 총 24~26
	- int[10000] : 배열 객체 헤더 (약 12~16바이트) + 배열 길이 정보(4바이트) + 실제 데이터 10000(4*10000) = 약 40KB
	- Integer[10000] or List<Integer> : 배열 자체의 껍데기 용량  + 1만개의 무작위의 위치에 생성된 Integer =  약 160KB 이상(12~16바이트 x 10000) + 극심한 메모리 파편화발생
- 저장소는 가벼운 원시 타입배열로 깔고, 흐름 제어하는 큐에는 객체 생성을 최소화 한다. 즉, 단일 데이터를 담을 때는 Wrapper 클래스(Integer)가 작은 배열(int[2])보다 가볍지만, 대량의 데이터를 저장할 때는 원시 타입 배열(int[])이 답도적으로 가볍고, 가비지 컬렉터(CG)를 편안하게 해준다.

## 그래프 탐색 & DP 아키텍처(BFS vs Dijkstra, DP)

1. BFS (너비 우선 탐색): 최단 거리
	- 목적: 모든 간선의 비용이 동일할 때, 도착지까지 가장 빨리 닿는 최단 경로 1개를 찾는다. (물결 퍼뜨리기)
	- 코어 엔진: Queue<Integer> (자바에서 ArrayDeque를 사용)
	- 절대 원칙 (메모리 폭발 방지): 큐에서 꺼낼 때가 아니라, **큐에 넣을 때 즉시 방문 도장을 찍는다.**
	- 객체 다이어트 기법:  무거운 객체 대신 int[] dist 배열 하나를 -1로 초기화하여, 방문 여부와 누적거리를 동시에 관리를 고려해본다.
2. Dijkstra (다익스트라): 최소 비용
	- 목적: 도로마다 통행료(가중치)가 다를 때, 가장 저렴한 최단 경로를 찾는다.
	- 코어 엔진: PriorityQueue<int[]>
		- 거리가 가장 짧은(싼) 노드부터 큐에서 먼저 튀어나오도록 정렬 기준을 세운다.
	- 절대 원칙(자연 갱신 스킵)
		- 큐에서 꺼낸 경로의 비용이, 이미 내 지도(dist 배열)에 기록된 최단 거리보다 비싸다면, 쓸모없는 데이터이므로 continue로 넘긴다.
3. DP (다이나믹 프로그래밍): 모든 경우의 수 (몇가지)
	- 목적: 특정 지점까지 갈 수 있는 경로의 총 가짓수나 누적된 최댓값을 구한다.
	- 코어 엔진: 1차원 또는 2차원 배열
	- 절대 원칙 (점화식과 메모이제이션):
		- 탐색(분신술)을 하지 않는다. 대신에 내 위치로 올 수 있는 과거의 값들을 더해 거시적은 룰만 세운다.
		- 계산된 값은 배열에 덮어써서 기록해 두고 재활용한다.
	- 백엔드 필수 고려사항:
		- 숫자가 기하급수적으로 커질 수 있으므로, 배열에 값을 더하는 즉시 실시간으로 %같은 모듈러 연산을 적용해 오버플로우를 차단한다.

## 이분 탐색 시 자료형에 주의한다

1. 이분 탐색으로 적절한 값을 구할 때 값이 int자료형(21억)을 쉽게 넘어갈 때가 있다. 이 때는 long형이나 BigInteger같은 자료형을 사용하도록한다.

## 이분탐색을 의심해야 하는 3가지 결정적 시그널

1. **탐색해야 할 정답의 범위(최댓값)가 1억(100,000,000)을 넘어갈 때**
- 보통 코딩 테스트 채점 서버(그리고 일반적인 자바 환경)에서 1초 = 약 1억 번의 연산이라고 한다. 그래서 만약 내가 찾아야할 값이 10억, 20억(`Integer.MAX_VALUE`), 혹은 그 이상이라면 단순 `for` 문으로 1부터 하나씩 찾으면 10~20초가 걸려서 무조건 Time Out가 발생한다. 그러니 찾는 범위가 1억이 넘어간다면 이분탐색을 의심해본다.
1. **변수 타입으로** **`long`** **을 요구 할 때**
- 문제 제한 사항에서 “최대 시간은 `10^14`이하입니다” 혹은 “정답은 `long` 타입으로 반환하세요” 같은 조건이 주어질 때가 있다. 자바에서는 int는 21억까지인데, 21억의 루프도 시간 촤가 나는데, 그걸 넘어선다고 가정하면은 무조건 이분 탐색(파라메트릭 서치)이다
1. **“최댓값의 최솟값” 또는 “최솟값의 최댓값”을 구하라고 할 때**
- 다음과 같이 특정 조건을 만족하는 선에서 가장 극한의 타협점(Mid)를 찾으라는 늬앙스라면 주정없이 이분탐색(파라메트릭 서치)를 사용한다
	- “모든 사람이 심사를 받는 데 걸리는 시간의 최솟값을 구하세요.”(입국심사)
	- “건널 수 있는 사람 수의 최댓값을 구하세요”(징검다리 건너기)

## 투 포인터 vs 슬라이딩 윈도우

1. 투 포인터 (마주보기)
- `while(left<right)` 가 메인 엔진이며, 양 끝 단에서 출발하여, 서로를 향해 좁혀오다가 교차하는 순간 탐색을 중요하다. 또한 <u>구간은 의미가 없고,</u> <u>`left`</u><u>와</u> <u>`right`</u> <u>가 가리키는 단일 값들의 조합</u>이 중요하다.
1. 슬라이딩 윈도우 (같은 방향 쫓아가기)
- `while(right < length)` 와 같은 조건절이 메인 엔진이 됩니다. `left`와 `right`  모두 배열의 시작점에서 출발하여 같은 방향으로 이동하며 두 포인터 사이의 <u>**구간(Window)안에 들어 있는 데이터의 상태**</u>가 핵심이다.

## 슬라이딩 윈도우의 포함 관계(중첩 루프)와 상태 머신(평면 구조 방식)

1. 중첩 루프 패턴(포함 관계)
- `right`를 전진시킨 후 내부에서 `while` 문으로 `left` 를 당겨오는 구조를 짠다면 → `while(right < length)` 나 `for` 문을 쓰면 된다.
1. 평면 구조 방식(상태 머신)
- `if (조건 만족하면 left 이동 else (아니면 right) 이동)` 처럼 평면적으로 조건을 분기한다면 → 무조건 `while(true)` 를 켜두고 명시적인 `break` 시점을 잡아주어야, `right` 가 끝난 후에도 `left` 가 남은 작업을 처리할 수 있다.

## 슬라이딩 윈동의 패턴 사용 예

1. 창문 크기가 고정 되어 있는가?  (예: “연속된 K일 동안의”)
- **고정 크기 패턴** → `left` 변수 없이 i와 i-k를 이용한 단일 `for` 문
1. 창문 크기가 유동적인가 (예: “합이 S 이상이 되는 가장 짧은 구간”)
- **중첩 루프 패턴** → `right` 전진 후 `while` 문으로 `left` 를 당기는 **중첩 루프 패턴**
1. 조건이 복잡하여 로직을 명확히 분기하고 싶은가?
- **상태 머신 패턴 →** `while (true)` 로 무한 루프를 돌리는 **상태 머신 패턴**
