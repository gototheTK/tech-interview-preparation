
## Q) 인터페이스와 추상 클래스의 본질적인 차이는 무엇인가요?


추상 클래스는 객체의 종류와 공통된 상태를 정의합니다. 자바에서는 다중 상속이 불가능합니다. 그에 반해 인터페이스는 객체의 역할을 정희하고, 상태를 가질 수 없으며 다중 상속이 가능합니다.


## Q) 구현체가 1개일 때도 무조건 인터페이스를 만드는 것이 좋은 설계인가요?


아닙니다. 과거 스프링에서는 AOP(@Transactional 등)를 적용하기 위해 JDK 동적 프록시를 사용했기 때문에 강제로 인터페이스가 필요했습니다. 하지만 Spring Boot 2.0부터 CGLIB(Code Generation Library)가 기본값으로 채택되면서, 클래스를 상속받아 프록시를 생성할 수 있게 되었습니다. 따라서 무분별한 인터페이스의 생성은 파일 탐색 비용만 높이므로, 확장이 불필요 하다면 클래스 단독으로 사용하는 것이 실용적입니다.


## Q) Spring 프록시 패턴과 템플릿 콜백 패턴(JdbcTemplate)의 결정적 차이는 무엇인가요?


**프록시 패턴(@Transactional)**은 타겟 코드를 수정하지 않고 겉에서 몰래 감싸(Wrapping) 부가 기능(트랜잭션 앞뒤 처리)을 추가합니다. 타겟 객체는 자신이 감싸진 줄도 모르며 수동적으로 동작합니다. 그에비해, **템플릿 콜백 패턴(JdbcTemplate)**은 변하지 않는 전체 흐름(DB 연결/해제 뼈대)을 미리 만들어 두고, 개발자가 능동적으로 템플릿을 호출하며 바뀔 부분(SQL 쿼리)만 파라미터(콜랙)로 던져 빈칸을 채우는 방식입니다.


## Q) 생성자 주입(Constructor Injection)을 권장하는 3가지 이유는 무엇인가요?

1. 불변성 보장: final 키워드를 사용하여 객체 생성 시점에 의존성을 고정할 수 있습니다.
2. 순환 참조 방지: A가 B를, B가 A를 참조할 경우, 컴파일 후 애플리케이션 기동(초기화) 시점에 스프링부트가 에러를 뿜어 시한폭탄을 막아줍니다.
3. 테스트 용이성: 스프링 컨테이너 없이도 순수 자바 코드로 가짜 객체(Mock)를 주힙해 단위 테스트를 하기 쉽습니다.

## Q) JPA에서 ‘save()’를 호출하지 않았는데도 UPDATE 쿼리가 나가는 이유가  무엇인가요?


트랜잭션 중에 영속성 컨텍스트가 엔티티 변경을 감지하여 자동으로 DB에 반영하는 '더티 체킹'을 하기때문입니다. JPA는 DB에서 엔티티를 처음 조회하여 1차 캐시인 영속성 컨텍스트에 저장할 때, 엔티티의 최초의 상태를 따로 복사하여 저장하는데 이를 스냅샷(Snapshot)이라고 합니다. 트랜랙션 중에 커밋(Flush)되는 시점에, 현재 엔티티 상태와 보관해둔 스냅샷을 비교하여 변경되었다면, JPA가 알아서 UPDATE 쿼리를 생성해 DB에 날려줍니다. 그렇기 때문에, 명시적으로 save()를 하지 않았더라도 엔티티를 변경하였다면 DB에 반영이 됩니다. 참고로, EAGER(즉시 로딩)을 사용하게되면, N+1이 바로 터지게됩니다. 그러므로 LAZY로 연관객체를 지연로딩을 시키고, Fetch Join으로 한꺼번에 가져오는게 국룰입니다.


## Q) N+1 문제의 발생 원인과 가장 확실한 해결책은 무엇인가요?


JPA에서 연관 관계를 지연 로딩(Lazy Loading)으로 설정하면, 연관된 객체는 실제 데이터가 아닌 프록시 객체로 채워지게 됩니다. 트랜잭션에서 이 후 연관된객체의 데이터를 요구할 때 마다 실제 데이터를 가져오기 위해 프록시 객체는 조회 쿼리를 실행하게 됩니다. 그리하여 추가 I/O 요청이 발생하여 DB에 부하를 주거나 DB 커넥션 풀에 병목현상을 일으키는 원인이 됩니다. 이를 해결하기위해 JPQL의 Fetch Join을 이용하여, 연관된 데이터를 한꺼번에 가져오게합니다.


## Q) Filter와 Interceptor의 차이점은 무엇이며, 각각 어떤 상황에 사용하나요?


가장 큰 차이는 실행 위치와 관리 주체입니다. 필터(Filter)는 DispatcherServlet에 요청이 도달하기 전, 웹 컨테이너(톰캣)에서 동작합니다. 스프링 컨텍스트와 무관하게 전체 HTTP 요청에 대한 **공통 처리**(인코딩 변환, 로깅, CORS 처리)를 할 때 주로 사용합니다. 인터셉터(Interceptor)는 DispatcherServlet을 통과한 후 스프링 컨테이너 내부에서 컨트롤러에 도달하기 전에 동작합니다. 스프링의 빈과 컨트롤러 정보에 접근할 수 있으므로, 특정 API의 세션 검증, **인가(Authorization)** 체크 등 비즈니스 로직과 밀접한 **처리를** 할 때 적절합니다.


참고로, 스프링 시큐리티는 이론상 서블릿의 Filter 단에서 동작해야 하지만, 톰캣은 **스프링이 관리하는 빈을** 인지하지 못해 스프링 시큐리티의 Filter(빈)를 직접 사용할 수 없습니다. 그러나 스프링은 필터로부터 처리를 위임받는 방식으로 이를 해결하는데, 미리 톰캣의 필터 단에 `DelegatingFilterProxy`라는 껍데기 필터를 등록해 놔서 **요청이 오면 가로채어 스프링 내부의** `FilterChainProxy`라는 스프링 빈에게 처리를 통째로 위임합니다.
