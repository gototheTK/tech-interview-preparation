
## Q) 인터페이스와 추상 클래스의 본질적인 차이는 무엇인가요?


추상 클래스는 객체의 종류와 공통된 상태를 정의합니다. 자바에서는 다중 상속이 불가능합니다. 그에 반해 인터페이스는 객체의 역할을 정희하고, 상태를 가질 수 없으며 다중 상속이 가능합니다.


## Q) 구현체가 1개일 때도 무조건 인터페이스를 만드는 것이 좋은 설계인가요?


아닙니다. 과거 스프링에서는 AOP(@Transactional 등)를 적용하기 위해 JDK 동적 프록시를 사용했기 때문에 강제로 인터페이스가 필요했습니다. 하지만 Spring Boot 2.0부터 CGLIB(Code Generation Library)가 기본값으로 채택되면서, 클래스를 상속받아 프록시를 생성할 수 있게 되었습니다. 따라서 무분별한 인터페이스의 생성은 파일 탐색 비용만 높이므로, 확장이 불필요 하다면 클래스 단독으로 사용하는 것이 실용적입니다.


## Q) Spring 프록시 패턴과 템플릿 콜백 패턴(JdbcTemplate)의 결정적 차이는 무엇인가요?


**프록시 패턴(@Transactional)**은 타겟 코드를 수정하지 않고 겉에서 몰래 감싸(Wrapping) 부가 기능(트랜잭션 앞뒤 처리)을 추가합니다. 타겟 객체는 자신이 감싸진 줄도 모르며 수동적으로 동작합니다. 그에비해, **템플릿 콜백 패턴(JdbcTemplate)**은 변하지 않는 전체 흐름(DB 연결/해제 뼈대)을 미리 만들어 두고, 개발자가 능동적으로 템플릿을 호출하며 바뀔 부분(SQL 쿼리)만 파라미터(콜랙)로 던져 빈칸을 채우는 방식입니다.


## Q) 생성자 주입(Constructor Injection)을 권장하는 3가지 이유는 무엇인가요?

1. 불변성 보장: final 키워드를 사용하여 객체 생성 시점에 의존성을 고정할 수 있습니다.
2. 순환 참조 방지: A가 B를, B가 A를 참조할 경우, 컴파일 후 애플리케이션 기동(초기화) 시점에 스프링부트가 에러를 뿜어 시한폭탄을 막아줍니다.
3. 테스트 용이성: 스프링 컨테이너 없이도 순수 자바 코드로 가짜 객체(Mock)를 주힙해 단위 테스트를 하기 쉽습니다.

## Q) JPA에서 ‘save()’를 호출하지 않았는데도 UPDATE 쿼리가 나가는 이유가  무엇인가요?


트랜잭션 중에 영속성 컨텍스트가 엔티티 변경을 감지하여 자동으로 DB에 반영하는 '더티 체킹'을 하기때문입니다. JPA는 DB에서 엔티티를 처음 조회하여 1차 캐시인 영속성 컨텍스트에 저장할 때, 엔티티의 최초의 상태를 따로 복사하여 저장하는데 이를 스냅샷(Snapshot)이라고 합니다. 트랜랙션 중에 커밋(Flush)되는 시점에, 현재 엔티티 상태와 보관해둔 스냅샷을 비교하여 변경되었다면, JPA가 알아서 UPDATE 쿼리를 생성해 DB에 날려줍니다. 그렇기 때문에, 명시적으로 save()를 하지 않았더라도 엔티티를 변경하였다면 DB에 반영이 됩니다. 참고로, EAGER(즉시 로딩)을 사용하게되면, N+1이 바로 터지게됩니다. 그러므로 LAZY로 연관객체를 지연로딩을 시키고, Fetch Join으로 한꺼번에 가져오는게 국룰입니다.


## Q) N+1 문제의 발생 원인과 가장 확실한 해결책은 무엇인가요?


JPA에서 연관 관계를 지연 로딩(Lazy Loading)으로 설정하면, 연관된 객체는 실제 데이터가 아닌 프록시 객체로 채워지게 됩니다. 트랜잭션에서 이 후 연관된객체의 데이터를 요구할 때 마다 실제 데이터를 가져오기 위해 프록시 객체는 조회 쿼리를 실행하게 됩니다. 그리하여 추가 I/O 요청이 발생하여 DB에 부하를 주거나 DB 커넥션 풀에 병목현상을 일으키는 원인이 됩니다. 이를 해결하기위해 JPQL의 Fetch Join을 이용하여, 연관된 데이터를 한꺼번에 가져오게합니다.


## Q) Filter와 Interceptor의 차이점은 무엇이며, 각각 어떤 상황에 사용하나요?


가장 큰 차이는 실행 위치와 관리 주체입니다. 필터(Filter)는 DispatcherServlet에 요청이 도달하기 전, 웹 컨테이너(톰캣)에서 동작합니다. 스프링 컨텍스트와 무관하게 전체 HTTP 요청에 대한 **공통 처리**(인코딩 변환, 로깅, CORS 처리)를 할 때 주로 사용합니다. 인터셉터(Interceptor)는 DispatcherServlet을 통과한 후 스프링 컨테이너 내부에서 컨트롤러에 도달하기 전에 동작합니다. 스프링의 빈과 컨트롤러 정보에 접근할 수 있으므로, 특정 API의 세션 검증, **인가(Authorization)** 체크 등 비즈니스 로직과 밀접한 **처리를** 할 때 적절합니다.


참고로, 스프링 시큐리티는 이론상 서블릿의 Filter 단에서 동작해야 하지만, 톰캣은 **스프링이 관리하는 빈을** 인지하지 못해 스프링 시큐리티의 Filter(빈)를 직접 사용할 수 없습니다. 그러나 스프링은 필터로부터 처리를 위임받는 방식으로 이를 해결하는데, 미리 톰캣의 필터 단에 `DelegatingFilterProxy`라는 껍데기 필터를 등록해 놔서 **요청이 오면 가로채어 스프링 내부의** `FilterChainProxy`라는 스프링 빈에게 처리를 통째로 위임합니다.


## Q) 스프링의 @Transactional 어노테이션을 붙이면 개발자가 직접 commit이나 rollback을 작성하지 않아도 트랜잭션이 적용되는 내부 원리(AOP)를 설명해 주세요.


스프링에서 @Transactional과 같은 어노테이션이 걸린 객체는 프록시라는 가짜 객체에 둘러싸이게 되고, 해당 메서드 역시 가짜 객체의 메서드에 부가기능과 함께 감싸지게 됩니다. 그래서 API를 호출하게 되면, 이 프록시 객체의 가짜 메서드가 실행되게 되고, 개발자가 직접 작성하지 않아도 try-catch, commit, rollback 등이 자동적으로 호출되어 트랜잭션이 관리되게 됩니다.


## Q) 스프링 빈(Bean)을 초기화할 때, 자바의 기본 생성자(Constructor) 내부에서 무거운 작업이나 외부 의존성 참조를 하지 않고 @PostConstruct를 사용하는 이유는 무엇인가요?


스프링의 빈 생명 주기에서 생성 시점과 의존성 주입 단계가 철저히 분리되어 있기 때문입니다. 그래서 빈을 생성하는 시점(생성자 내부)에서 외부 API 연결이나 DB 초기 데이터 로드 등과 같은 초기화 작업을 하게 된다면, 필요한 의존성 주입을 받지 못한 상태이므로 NullPointerException이 발생하게 됩니다. 따라서 객체의 생성과 초기화를 분리하여야 하고, 주입이 완료된 후 안전하게 초기화 작업을 진행하기 위해 @PostConstruct 어노테이션을 사용해야 합니다.


## Q) @Transactional으로 선언된, 메서드에서 JPA 처리와 Spring JDBC 처리를 동시에 하였습니다. 이 둘은 같은 트랜잭션으로 묶이나요?


하나의 트랜잭션으로 묶여 원자성(Atomicity)를 보장합니다. @Transactional을 JPA의 영속성 컨텍스트로 오해하지만, 스프의 트랜잭션 매지너는 내부적으로 동일한 DataSource를 통해 발급받은 하나의 Connection을 `TransactionSynchronizationManager` 에 동기화 시킵니다. 따라서 영속성 컨텍스트를 거치는 JPA의 로직이나, Connection을 직접 가져다쓰는 `JdbcTemplate` 로직 모두 동일한 트랜잭션 범위 안에서 안전하게 커밋되거나 롤백됩니다.


## Q) 과거의 레거시 스프링(WAR 배포)과 비교하여, 스프링 부트(Spring Boot) 환경에서 내장 톰캣(Embedded Tomcat)이 구동되는 원리와 외부 HTTP 요청을 처리하는 트래픽 흐름에 대해 설명해주세요.



스프링부트가 실행되면 스프링 컨테이너가 먼저 만들어지고 생성자 주입 등 빈의 생성과 초기화를 진행합니다. 스프링의 빈과 초기화가 완료되면, TomcatServletWebServerFactory라는 팩토리 빈을 사용하여 웹 컨테이너인 내장 톰캣 객체를 생성합니다. 이 때, 스프링 컨테이너는 프론트 컨트롤러로서 톰캣이 사용할 서블릿으로 DispatcherServlet이라는 빈을 주입한 후에 서버 부팅을 완료합니다. 이 후 외부에서 HTTP 요청이 오면 만들어진 톰캣이 이를 해석해서 HttpServletRequest라는 객체로 만들어, DispatcherServlet으로 전달하고, DispatcherServlet은 적절한 Controller에게 전달하여 처리하도록 합니다.


## Q) 운영체제의 소켓(Socket)부터 자바의 서블릿(Servlet)까지, 클라이언트의 웹 요청이 들어왔을 때 데이터가 전달되고 변환되는 전체 흐름을 설명해 주세요.



데이터의 전달과 흐름은 크게 OS(소켓), 웹 컨테이너(톰캣), 자바 애플리케이션(서블릿) 3단계로 이루어져 있습니다. 먼저 OS(커널)의 TCP/IP 계층인 소켓이 세그먼트를 이어 붙여 바이트 스트림으로 만들어서 톰캣에게 전달해 줍니다. 그러면 톰캣은 이 바이트 스트림을 해석하여 자바 애플리케이션이 다루기 쉽도록 `HttpServletRequest`와 `HttpServletResponse` 객체로 만들어서 스프링과 같은 자바 애플리케이션에게 전달하여 줍니다. 그러면 스프링의 `DispatcherServlet`은 이 두 객체를 인자로 받아서 적절한 컨트롤러를 찾아 처리하도록 합니다. 참고로, 스프링의 `DispatcherServlet`은 Java EE(현 Jakarta EE)에서 선언한 표준 HTTP 처리 인터페이스인 서블릿(Servlet)을 상속하여 구현되었습니다.


## Q) 네트워크 통신 시 데이터를 계층(Layer)에 따라 부르는 명칭(PDU, Protocol Data Unit)이 다릅니다. 상위 계층부터 하위 계층으로 내려가면서 데이터의 명칭이 어떻게 변하는지 설명해 주세요.


OSI 7계층에서는 데이터를 캡슐화하는 위치에 따라 부르는 명칭(PDU)이 다릅니다. 응용 계층에서는 메시지 혹은 바이트 스트림으로 불리고, 포트 번호를 붙이는 전송 계층에서는 TCP를 사용할 경우 세그먼트, UDP를 사용할 경우에는 데이터그램으로 불립니다. IP 주소를 붙이는 네트워크 계층에서는 패킷으로 불리고, MAC 주소를 붙이는 데이터 링크 계층에서는 프레임이라고 불립니다. 마지막으로 물리 계층에서의 데이터는 날것의 비트(Bit)로 불립니다.


## Q) 과거 레거시 스프링(WAR)에서 사용하던 '부모/자식 컨텍스트' 분리 구조의 진짜 의미는 무엇이며, 현대의 스프링 부트(Spring Boot)에서는 이 구조가 어떻게 변화했나요?


과거 스프링 프레임워크에서는 웹전용, 앱전용등 DispatcherServlet이 여러개였기 때문에, 공통 비지니스 로직 전용인 부모 컨텍스트(Service, Repository 등)와 API 처리 전용인 자식 컨텍스트(Controller)으로 나뉘어져 있었습니다. 부모는 자식을 참조할 수 없으며, 자식은 부모를 참조 할 수있다는 단방향 참조 범위(가시성)때문에 부모 컨텍스트와 자식 컨텍스트로 불립니다. 실제로, Controller가 실행되어 로직을 처리하기 위해 Service나 Repository를 참조할 때, 부모 컨텍스트를 참조합니다. 현대의 스프링 부트에서는 컨텍스트가 따로 나뉘어져 있지않고, 하나의 컨텍스트로 모든 빈을 관리합니다. 그래서 스프링 부트는 DispatcherSerlvet이 하나입니다. 왜냐하면, DispatcherServlet이 하나의 컨텍스트에서 Controller를 찾기때문입니다.


## Q) 스프링 시큐리티에서 인증(Authentication)과 인가(Authorization)의 차이는 무엇이며, 이 보안 검사를 비즈니스 로직(DispatcherServlet) 도달 전에 처리하기 위해 스프링은 톰캣(웹 서버)과 어떻게 협력하나요?


보안에서 인증은 신원을 확인하여 출입증을 발급하는 절차이고, 인가는 신원이 확인된 유저의 권한을 식별하여 통제하는 절차입니다. 이러한 절차들은 무거운 I/O작업이 존재하는 비즈니스 단(DispatcherServlet 및 Controller)에 도달하기 전에 처리하여 '빠른 실패(Fail-Fast)'를 유도하는 것이 좋습니다. 그래서 톰캣과  같은 웹 컨테이너 단의 Filter를 사용하여 보안 절차를 처리하는 것이 바람직합니다.


그러나 톰캣은 스프링 빈을 알지 못하므로, 스프링 시큐리티는 미리 톰캣에 DelegatingFilterProxy라는 대리 객체를 미리 등록해 둡니다. 그러면 매 요청마다 이 필터가 실행되고, 이 필터는 스프링 컨테이너의 보안 담당 빈(FilterChainProxy)을 찾아서 요청 넘겨서, 보안 절차를 처리하도록 합니다. 참고로, 보안 절차를 수행하는 보안 필터인증, 인가, 예외 처리를 담당하는 ExceptionTranslationFilter 등)들은 FilterChainProxy안에 리스트(List)형태의 멤버변수로 존재합니다. 그래서 보안 절차가 순차적으로 실행되고, 보안 절차를 통과한 요청은 DispatcherServlet에게 전달되어 해당하는 컨트롤러에게 전달되고, 통과되지 못한 요청들은 예외를 터뜨리고, HttpServletResponse에 실패 데이터(401/403)를 담아 클라이언트에게 전달됩니다.


## Q) 리다이렉트와 포워딩의 차이가 무엇인가요?


리다이렉트와 포워드의 차이점은 URL변화 여부와 요청 객체의 동일 여부에 있습니다. 리다이렉트는 웹 컨테이너를 벗어나 클아이언트에 도달한 뒤 다시 새로운 요청을 보내는 방식입니다. 그래서 URL이 변환되고, 요청에 대해서는 절대 동일하지 않고, 똑같은 상태 정보를 보장하지 못합니다. 그에 비해 포워드는 스프링 컨테이너 안에서 다른 컨트롤러를 찾아 요청 객체를 전달해 줍니다. 그래서 URL 변환이 없고, 요청에 대해서 동일할 수 있고, 똑같은 상태 정보를 보장할 수 있습니다.
