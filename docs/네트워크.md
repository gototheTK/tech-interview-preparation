
## Q) 분산 서버 환경에서 Session 대신 JWT를 사용하는 핵심적인 이유는?


상태(Stateful)방식인 Session을 사용하면, 서버의 메모리나 외부 저장소(Redis)에 장부처럼 보관해야 하므로, 스케일 아웃(서버 증설) 시 세션 동기화 문제와 네트워크 I/O 병목이 발생합니다. 그에비해 무상태(Stateless)방식인 JWT를 이용하면, 클라이언트가 토큰을 보관하며, 서버는 DB를 뒤질 필요가 없이 비밀키를 이용한 복호화 연산만으로 여부를 검증하므로 확장성이 압도적으로 좋습니다.


## Q) TCP 3-way Handshake 과정에서 악의적인 클라이언트가 SYN만 보내고 ACK를 보내지 않는다면 서버에는 어떤 일이 발생하나요? (SYN Flooding)


서버는 클라이언트로부터 요청을 받기 위해  통신 포트당 메모리에 공간을 할당하여 SYN Backlog Queue라는 대기열 큐를 사용하는데, 만약 악의적인 해커가 ACK를 보내지 않고, 계속 SYN만을 보내어 수 만건의 SYN을 보내게 된다면 서버의 대기열큐는 꽉 차게되고, 정상적인 클라이언트의 요청에 처리를 할 수가없고 튕겨내버리는 서비스 거부(Dos) 현상이 일어나게 됩니다. 이건 마치 DB 커넥션 풀이 고갈되어 장애가 나는 것과 같은 병목 현상입니다. 참고로, 서버가 SYN+ACK보내고나서 일정시간 동안 ACK를 못받는다면(Timeout) 해당 SYN을 삭제해버립니다. (Timeout Drop)


## Q) HTTPS는 대칭키(비밀번호)를 교환할 때 해커가 개로채지 못하도록 비대칭키 암호화 방식을 어떻게 활용하나요?


클라이언트가 인증기관으로부터 서버의 주소를 보증을 받는다면 서버에게 통신 요청을 보냅니다. 그러면 서버는 공개키를 클라이언트에게 보내줍니다. 그러면 클라이언트는 대칭키를 생성하여 공개키로 암호화하여 서버에게 보내줍니다. 그걸 받은 서버는 개인키로 복호화하여 대칭키를 받고 사용하여 클라이언트와 통신을 선언하고 데이터를 대칭키로 암호화, 복호화를 하며 주고 받습니다. 이 때, 대칭키가 사용하는 알고리즘은 서버도 알고 있어야합니다.


## Q) HTTP 메서드에서 GET, PUT은 멱등성이 보장되지만 POST는 보장되지 않는다는 것은 어떤 의미인가요?


멱등성은 동일한 요청에 대해 한 번만 보내는것과 여러번 보내는것에 대해 동일한 결과를 서버가 보내는 성질입니다. 이를 HTTP통신에서는 GET과 PUT으로 구현하였는데, GET과 PUT 모두 동일한 요청에 대해 여러번 요청을 하여도, 동일한 결과를 보내주기 때문입니다. GET은 똑같은 조회, PUT은 수정 및 덮어쓰기로 결과는 같습니다. 다만, POST는 똑같은 요청에 대해 결과를 다르게 보여줄 수 있는데, 가령 결제 버튼이나 글 쓰기 버튼을 여러번 클릭하게되면 중복 결제나 중복 글쓰기의 현상이 발생하게 됩니다. 즉, 최초의 결과와 달라져 멱등성이 보장되지 않습니다.


## Q) HTTP의 무상태성(Stateless)을 극복하기 위해 쿠키(Cookie)는 어떻게 동작하나요?


HTTP는 기본적으로 무상태(Stateless)성가지며 클라이언트를 식별할 수 없습니다.  이를 해결하기 위해 만들어진게 쿠키라는 데이터입니다. 원리는 서버가 응답으로 헤더에 Set-Cookie를 넣어서 전송하면, 클라이언트는 이 Set-Cookie의 값을 잠시 저장했다가, 요청 시에 헤더에 Cookie의 값으로 명시하여 보냅니다. 그러면 서버는 이 값을 통해 클라이언트를 식별합니다. 쿠키는 최대 4kb의 크기를 가지며, 기본적으로 세션 쿠키로서 브라우저가 종료되면 사라집니다. 대신에 기한을 정해놓으면 디스크에 저장되는 영속쿠키가 됩니다. IP와 포트로 클라이언트를 식별하지 않는 이유는 통신 포트는 브라우저에서 페이지를 이동할 때마다 수시로 바뀌고, IP는 공인아이피로 전달되기 때문에 클아이언트를 식별할 수가 없습니다. 톰캣은 JSESSIONID이라는 난수 값으로 쿠키에 값을 보내 클라이언트를 식별합니다.


## Q) 프론트엔드와 분리된 환경에서 발생하는 CORS(Cross-Origin Resource Sharing) 에러의 원인과 해결책은 무엇인가요?


CORS(Cross-Origin-Resource-Sharing)는 페이지를 받은 서버와 페이지에서 요청을 하는 서버의 차이를 허용하는 보안 절차입니다. 브라우저는 동일 출처라는 SOP 규칙을 따르는데, 페이지를 받은 서버와 요청하는 서버 간에 차이가 나는 것을 보통 허락하지 않습니다. 그렇기 때문에 출처 간의 차이가 난다면, 브라우저는 응답 서버에 OPTIONS 메서드로 사전 요청(Preflight)을 하여 출처를 확인받습니다. 그러면 응답 서버는 헤더의 Access-Control-Allow-Origin에 허용할 페이지 서버의 주소를 값으로 명시하여 확인시켜줍니다. 응답을 받은 브라우저는 이 값과 자신이 페이지를 받은 서버의 주소를 대조하여, 틀리면 SOP 위반 오류를 띄우고, 맞으면 비로소 응답 서버(백엔드 서버)와의 API 통신을 허용합니다.


## Q) 프론트와 백엔드가 분리된 환경에서 세션 쿠키(인증 정보)를 주고받을 때 발생하는 CORS 문제 원인과, 이를 해결하기 위한 백엔드 서버의 정확한 헤더 설정(Credentials, Origin Max-Age)에 대해 설명해주세요.


브라우저는 보안(SOP)을 위해 출처가 다른 서버로 API 요청을 보낼 때 기본적으로 쿠키(인증 정보)를 포함하지 않으며, 프론트엔드에서 이를 강제로 포함해 보내더라도 서버가 명시적으로 허락하지 않으면 CORS 에러를 발생시켜 데이터를 차단합니다.


이 문제를 해결하고 안전하게 쿠키를 주고받기 위해서는 프론트엔드와 백엔드 양측의 설정이 필요합니다.

1. 프론트 엔드: 요청 시 `withCredentials: true` 옵션을 설정하여 쿠키 컨송 의지를 밝힙니다.
2. 백엔드 (헤더 설정): 브라우저의 사전 요청(Preflight) 및 본 응답에 대해 다음 헤더를 설정하여 브라우저를 안심시켜야 합니다.
	- `Access-Control-Allow-Credentials: true` : 서버가 쿠키 등 인증 정보 수신을 명시적으로 허용함을 뜻합니다.
	- `Access-Control-Allow-Origin: 허용할 프론트 엔드의 정확한 주소` : (주의점) Credentials를 tru로 설정할 때, Origin을 `*` (와일드 카드)로 지정하면 브라우저가 보안 위반으로 간주하여 CORS 에러를 띄우므로 반드시 정확한 출처를 명시해야 합니다.
	- `Access-Control-Max-Age: 시간(초)` : 브라우저가 사전 요청(Preflight)의 결과를 지정된 시간 동안 메모리에 캐싱하여, 매 요청마다 OPTIONS 메서드가 날아가는 네트워크 오버헤드를 줄여줍니다.

## Q) 실무에서 해커가 자바스크립트(document.cookie)를 악용하여 사용자의 세션 아이디(JSESSIONID)를 탈취하는 XSS(교차 사이트 스크립팅) 공격을 방허하기 위해, 쿠키를 구울 때 반드시 설정해야 하는 보안 플래그는 무엇인가요?


HttpOnly입니다. 이 설정을 하면, 브라우저의 자바스크립트가 접근할 수가 없습니다. 그렇게되면, XSS 공격을 통한 세션 탈취를 원천 차단합니다.


## Q) 낙관적 락(Optimistic Lock)을 적용한 상품 주문 로직에서, 버전(@Version) 충돌이 발생하여 ‘ObjectOptimisticLockingException’ 예외가 터졌습니다. 애플리케이션 레벨에서 개발자가 취해야 할 올바른 후속 조치는 무엇인가요?


낙관적 락은 충돌 시 예외를 던지는 것까지만 책임지므로, 이후 복구(재조회 및 재시도)는 온전히 개발자의 비즈니스 로직에 달려있습니다. 즉, 예외를 잡아서(Catch) 최대 3회 정도 조회를 다시 하고 업데이트를 시도하는 재시도(Retry) 로직을 구현하거나, 사용자에게 ‘다시 시도해주세요’ 메시지를 보여줍니다.


오답) 런타임 중에 트랜잭션 격리 수준을 동적으로 SERIALIZABLE로 바꾸는 것은 불가능에 가깝고, 성능을 심각하게 저하시킵니다.


## Q) 네트워크 통신에서 TCP Keep-Alive(L4)와 HTTP Keep-Alive(L7)의  핵심적인 차이와 목적을 올바르게 설명한 것은 무엇인가요?


TCP Keep-Alive는 상대방이 죽었는지(Dead Peer) 상태인지 확인하여, 좀비 소켓을 정리하기 위한 방어 목적이고, HTTP 레벨의 Keep-Alive는 매번 3-Way Handshake를 하는 오버헤드를 줄여주는 HTTP 1.1부터 도입된 파이프라이닝 기술입니다.


동작원리

1. 클라이언트가 첫 요청 헤더에 Connection: Keep-alive를 담아서 보냅니다. (HTTP/1.1부터는 안 적어도 기본으로 적용됩니다.)
2. 서버가 TCP/IP 연결을 계속 안 끊겠다는 선언으로 `Keep-Alive: timeout=대기 시간, max=최대 요청 개수` 같은 조건을 달아 응답해 줍니다.
3. 클라이언트는 받은 조건대로 TCP/IP연결을 재사용합니다.
4. 더 이상 요청할 게 없고 타임아웃(timeout) 시간이 지나면, 4-Way-Handshake를 통해 통신을 종료합니다.

참고) Keep-Alive의 최대 요청 개수(max)를 초과해버리면, 4-Way-Handshake로 TCP/IP의 연결을 종료합니다. 통신을 계속하려면 새로운  TCP/IP연결을 생성하여야 합니다. 이는 소켓과 같은 자원을 한 클라이언트가 독점하지 못하게 하기 위함입니다.


## Q) 과거의 세션(Session) 방식이 가진 한계점과, 이를 해결하기 위해 등장한 JWT의 핵심 아키텍처는 무엇인가요?


과거에는 `JSESSIONID`라는 유저 식별 키를 발급하여, 유저의 상태 정보와 인가 정보를 서버 메모리의 '세션'이라는 장부에 기록(Stateful)하였습니다. 그래서 유저가 많아질수록 서버 메모리에 부담이 많이 갔고, 특히 서버를 증설(Scale-Out)할 경우 로드밸런서에 의해 '인증된 서버'가 아닌 '인증되지 않은 서버'로 요청이 라우팅되면 세션에 유저 정보가 없어 인증이 풀려버리는 사태가 발생하여 서버 확장에 큰 어려움이 있었습니다.
이를 해결하기 위해 해시와 '무상태성(Stateless)'을 이용한 JWT가 도입되었습니다. JWT는 서버에 아무것도 저장하지 않고, 서명과 인가 정보를 토큰에 담아 클라이언트에게 발급한 뒤 매 요청마다 비밀키(Secret Key)로 이 정보들이 조작되었는지 확인하는 원리입니다. 덕분에 서버 메모리의 부담을 크게 줄일 수 있게 됐고, 서버 증설의 어려움도 완벽하게 덜어낼 수 있었습니다.


## Q) JWT의 내부 구조와 해커의 토큰 조작을 방어하는 무결성 검증 원리는 무엇인가요?


`[Payload 데이터(유저의 인가 정보 등 데이터)] + [서명(Signature)]` 형식입니다. 서버는 토큰 발급 시 서버들만 공유하고 있는 비밀키와 `[Header 데이터 + Payload 데이터]`를 결합하여 해싱(Hashing)해 서명을 만들고, 이를 합쳐 완전한 토큰으로 유저에게 발급합니다. 유저는 이 토큰을 저장해 두었다가 매 요청마다 돌려보내고, 서버는 자신의 비밀키와 토큰의 `[Header + Payload]`를 다시 결합하여 직접 해싱해 본 뒤 기존 서명과 대조합니다. 이는 해시 함수의 복호화가 불가능한 '단방향성'과, 입력 데이터가 1비트라도 바뀌면 결과값이 완전히 바뀌어버리는 '눈사태 효과(Avalanche Effect)' 특징을 이용한 원리입니다. 덕분에 해커가 Payload를 1글자라도 조작하면 서명이 절대 일치하지 않게 되어, 서버가 위조 여부를 즉시 확인하고 차단할 수 있습니다.
