
## Q) 분산 서버 환경에서 Session 대신 JWT를 사용하는 핵심적인 이유는?


상태(Stateful)방식인 Session을 사용하면, 서버의 메모리나 외부 저장소(Redis)에 장부처럼 보관해야 하므로, 스케일 아웃(서버 증설) 시 세션 동기화 문제와 네트워크 I/O 병목이 발생합니다. 그에비해 무상태(Stateless)방식인 JWT를 이용하면, 클라이언트가 토큰을 보관하며, 서버는 DB를 뒤질 필요가 없이 비밀키를 이용한 복호화 연산만으로 여부를 검증하므로 확장성이 압도적으로 좋습니다.


## Q) TCP 3-way Handshake 과정에서 악의적인 클라이언트가 SYN만 보내고 ACK를 보내지 않는다면 서버에는 어떤 일이 발생하나요? (SYN Flooding)


서버는 클라이언트로부터 요청을 받기 위해  통신 포트당 메모리에 공간을 할당하여 SYN Backlog Queue라는 대기열 큐를 사용하는데, 만약 악의적인 해커가 ACK를 보내지 않고, 계속 SYN만을 보내어 수 만건의 SYN을 보내게 된다면 서버의 대기열큐는 꽉 차게되고, 정상적인 클라이언트의 요청에 처리를 할 수가없고 튕겨내버리는 서비스 거부(Dos) 현상이 일어나게 됩니다. 이건 마치 DB 커넥션 풀이 고갈되어 장애가 나는 것과 같은 병목 현상입니다. 참고로, 서버가 SYN+ACK보내고나서 일정시간 동안 ACK를 못받는다면(Timeout) 해당 SYN을 삭제해버립니다. (Timeout Drop)


## Q) HTTPS는 대칭키(비밀번호)를 교환할 때 해커가 개로채지 못하도록 비대칭키 암호화 방식을 어떻게 활용하나요?


클라이언트가 인증기관으로부터 서버의 주소를 보증을 받는다면 서버에게 통신 요청을 보냅니다. 그러면 서버는 공개키를 클라이언트에게 보내줍니다. 그러면 클라이언트는 대칭키를 생성하여 공개키로 암호화하여 서버에게 보내줍니다. 그걸 받은 서버는 개인키로 복호화하여 대칭키를 받고 사용하여 클라이언트와 통신을 선언하고 데이터를 대칭키로 암호화, 복호화를 하며 주고 받습니다. 이 때, 대칭키가 사용하는 알고리즘은 서버도 알고 있어야합니다.


## Q) HTTP 메서드에서 GET, PUT은 멱등성이 보장되지만 POST는 보장되지 않는다는 것은 어떤 의미인가요?


멱등성은 동일한 요청에 대해 한 번만 보내는것과 여러번 보내는것에 대해 동일한 결과를 서버가 보내는 성질입니다. 이를 HTTP통신에서는 GET과 PUT으로 구현하였는데, GET과 PUT 모두 동일한 요청에 대해 여러번 요청을 하여도, 동일한 결과를 보내주기 때문입니다. GET은 똑같은 조회, PUT은 수정 및 덮어쓰기로 결과는 같습니다. 다만, POST는 똑같은 요청에 대해 결과를 다르게 보여줄 수 있는데, 가령 결제 버튼이나 글 쓰기 버튼을 여러번 클릭하게되면 중복 결제나 중복 글쓰기의 현상이 발생하게 됩니다. 즉, 최초의 결과와 달라져 멱등성이 보장되지 않습니다.


## Q) HTTP의 무상태성(Stateless)을 극복하기 위해 쿠키(Cookie)는 어떻게 동작하나요?


HTTP는 기본적으로 무상태(Stateless)성가지며 클라이언트를 식별할 수 없습니다. 이를 해결하기 위해 만들어진게 쿠키라는 데이터입니다. 원리는 서버가 응답으로 헤더에 Set-Cookie를 넣어서 전송하면, 클라이언트는 이 Set-Cookie의 값을 잠시 저장했다가, 요청 시에 헤더에 Cookie의 값으로 명시하여 보냅니다. 그러면 서버는 이 값을 통해 클라이언트를 식별합니다. 쿠키는 최대 4kb의 크기를 가지며, 기본적으로 세션 쿠키로서 브라우저가 종료되면 사라집니다. 대신에 기한을 정해놓으면 디스크에 저장되는 영속쿠키가 됩니다. IP와 포트로 클라이언트를 식별하지 않는 이유는 통신 포트는 브라우저에서 페이지를 이동할 때마다 수시로 바뀌고, IP는 공인아이피로 전달되기 때문에 클아이언트를 식별할 수가 없습니다. 톰캣은 JSESSIONID이라는 난수값으로 쿠키에 값을 보내 클라이언트를 식별합니다.


## Q) 프론트엔드와 분리된 환경에서 발생하는 CORS(Cross-Origin Resource Sharing) 에러의 원인과 해결책은 무엇인가요?


CORS(Cross-Origin-Resource-Sharing)는 페이지를 받은 서버와 페이지에서 요청을 하는 서버의 차이를 허용하는 보안 절차입니다. 브라우저는 동일 출처라는 SOP 규칙을 따르는데, 페이지를 받은 서버와 요청하는 서버 간에 차이가 나는 것을 보통 허락하지 않습니다. 그렇기 때문에 출처 간의 차이가 난다면, 브라우저는 응답 서버에 Preflight이라는 사전 요청을 하여 출처를 확인받습니다. 그러면 응답 서버는 헤더의 Access-Control-Allow-Origin에 허용할 페이지 서버의 주소를 값으로 명시하여 확인시켜줍니다. 응답을 받은 브라우저는 이 값과 자신이 페이지를 받은 서버의 주소를 대조하여, 틀리면 SOP 위반 오류를 띄우고, 맞으면 비로소 응답 서버(백엔드 서버)와의 API 통신을 허용합니다.
