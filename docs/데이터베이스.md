
## Q) 트랜잭션의 ACID 중 원자성과 지속성, 그리고 Undo/Redo Log의 역할은?

- 원자성 (Atomicity): 트랜잭션 내의 작업이 모두 완벽히 성공하거나 모두 실패해야합니다. (All or Nothing)
- 지속성 (Durability): 커밋된 트랜잭션의 결과는 시스템 장애(정전 등)가 발생해도 영구적으로 보존되어야 합니다.
- Undo Log: 트랜잭션 실패시, 변경 이전의 ‘체크포인트’ 시점으로 롤백(되돌리기)하여 원자성을 보장합니다.
- Redo Log: 메모리 버퍼(더피 페이지)에만 커밋되고 디스크에 쓰이지 않은 상태에서 장애가 났을 때, 재실행(복구)하여 지속성을 보장합니다.

## Q) 커넥션 풀(HikariCP)을 사용하는 이유와, 외부 API 지연시 발생하는 연쇄 장애는?

- 사용이유
	- DB와의 TCP/IP 연결/해제 과정이 매우 무겁고 느리기 때문에, 미리 소수의 파이프(커넥션)을 연결해 두고 빌려쓰는 방식을 취합니다. 이를 HikariCP가 제공하는 프레임워크이고, 이상적인 풀 사이즈는 (DB의 CPU 코어 수 * 2)  + 1(SSD사용 시)입니다.
- 연쇄 장애 시나리오
	- @Transational 내부에서 외부 API 호출(예: 결제 3초 대기)이 지연되면, 해당 톰켓 스레드가 DB 커넥션을 쥔 채 대기합니다. 이 트래픽이 몰리면 커넥션 풀이 고갈되고, 결국 요청을 받은 톰캣 스레드 풀마저 모두 멈춰 서서 전체 시스템이 먹통이 됩니다.

## Q) 왜 하필 DB는 HashTable이 아닌 B-Tree 인덱스계열을 표준으로 채택했나요?

- 가장 큰 이유는 데이터베이스의 핵심 탐색 패턴인 **범위 검색(Range Scan)**과 **정렬(Order By)** 때문입니다.
- 해시 테이블은 탐색 시간 복잡도가 $O(1)$로 매우 빠르지만, 입력값이 조금만 달라도 완전히 엉뚱한 해시값을 반환하므로 데이터의 대소 관계(순서)가 파괴되어 범위 검색이 불가능합니다.
- 반면, B-Tree는 데이터가 항상 **정렬된 상태**를 유지하는 트리 구조이므로, 부등호(`<`, `>`)나 `BETWEEN`을 활용한 순차적인 범위 검색을 매우 빠른 속도로 처리할 수 있습니다.

## Q) 클러스터드(Clustered) 인덱스와 논클러스터드(Non-Clustered) 인덱스의 결정적 차이는 무엇인가요? 영어사전과 일반 전공 서적의 맨 뒤 찾아보기(색인)을 예시로 들어주세요.

- 두 인덱스의 결정적 차이는 **'인덱스의 정렬 순서와 실제 데이터의 물리적 정렬 순서가 일치하는가?'**입니다.
- **클러스터드 인덱스 (영어 사전):** 실제 데이터 자체가 인덱스 순서대로 물리적 정렬이 되어 있습니다. 물리적인 정렬은 단 한 가지 방향으로만 가능하므로, **테이블당 오직 1개**만 생성할 수 있습니다. (주로 PK에 적용)
- **논클러스터드 인덱스 (전공서적 맨 뒤 색인):** 실제 데이터의 물리적 위치는 그대로 둔 채, 별도의 공간에 인덱스(목차)와 포인터만 생성합니다. 물리적 정렬과 무관하므로 **테이블당 여러 개**를 만들 수 있습니다.

## Q) 인덱스를 무진장 많이 걸면 안되는 이유가 무엇인가요?


인덱스 역시 결국 하드디스크에 별도로 저장되어 공간을 차지하는 **물리적인 데이터(B-Tree 파일)**입니다. 따라서 인덱스가 많아지면 `SELECT` 조회는 빨라지지만, 새로운 데이터를 삽입(`INSERT`), 수정(`UPDATE`), 삭제(`DELETE`)할 때마다 테이블뿐만 아니라 **모든 인덱스 트리를 찾아가서 갱신해야 하므로 디스크 쓰기(I/O) 부하가 폭증**합니다. 특히, 데이터가 꽉 찬 B-Tree 노드에 새로운 데이터를 억지로 끼워 넣어야 할 때 노드가 두 개로 쪼개지는 **'페이지 분할(Page Split)'**이 발생하며, 이는 엄청난 시스템 부하를 유발합니다. 따라서 조회가 잦은 핵심 컬럼에만 최소한으로 인덱스를 생성해야 합니다.


## Q) 옵티마이저가 비슷한 데이터에 똑같은 쿼리를 실행하는데, 어느 때는 0.1초 만에 처리되고, 어느 때는 10초가 걸리는 경우가 있습니다. 이런 현상이 왜 발생할 수 있는지, 옵티마이저가 실행 계획을 세울 때 참고하는 **'통계 정보(Statistics)'**라는 키워드와 엮어서 설명해 보세요.


옵티마이저가 때때로 오래된 통계 정보를 바탕으로 잘못된 실행 계획을 세우기 때문입니다.  통계 정보는 데이터 건수뿐만 아니라 데이터의 분포도(히스토그램) 기록합니다. 만약 전체 데이터 건수는 비슷한데, 특정 값의 분포만 갑자기 변해버리면 통계 정보가 반영이 안됩니다. 그러니 옵티마이저는 이전의 통계정보를 가지고 인덱스 스캔을 강행하게 되고 성능 저하를 유발하게 됩니다.


## Q) 옵티마이저가 인덱스가 존재함에도 불구하고, 이를 무시하고 의도적으로 Full Table Scan을 선택하는 경우가 있습니다. 어떤 상황에서 이런 선택을 하며, 왜 그게 물리적으로 더 빠른지 랜덤I/O와 순차I/O의 차이점을 들어 설명해주세요.


옵티마이저는 읽어야 할 데이터가 테이블 전체의 약 20%~25%를 넘어가서 손익분기점을 넘긴다고 판단한다면, 인덱스를 타는것보다 Full Table Scan을 실행하는 것을 선택합니다. 참고로, 인덱스 스캔은 디스크의 여러 곳을 흩어져서 읽는 랜덤I/O를 유발하고, Full Table Scan은 디스크를 연속해서 뭉텅이로 읽어 들이는 순차 I/O를 유발합니다.


## Q) 쿼리 앞에 EXPLAIN 명령어를 붙여 실행 계획을 뜯어보았는데, 다음과 같은 결과가 나왔습니다. 이 두 가지 지표가 의미하는 바가 각각 무엇인지 구체적으로 설명하고, 이를 해결하기 위해 어떤 조치가 필요한지 설명해주세요!

- `type : ALL`
- `Extra : Using filesort`

type : ALL은 인덱스를 타지 못하고 Full Table Scan을 했다는 의미입니다. 조건절에 자주 사용되는 컬럼에 인덱스를 생성하여 이를 해결합니다.


Extra : Using filesort는 ORDER BY 절을 처리할 적절한 인덱스가 없어서, DB 엔진이 데이터를 메모리의 버퍼내지는 디스크에 올려 놓고 직접 정렬 연산을 수행했다는 경고입니다. 트래픽이 몰리면 DB의 CPU 사용률이 급증하여 장애를 유발하므로, 정렬하려는 컬럼들에게 인덱스를 생성하여 이를 해결합니다.


## Q) 트랜잭션 이상 현상 중 Non-Repeatable Read와 Phantom Read의 결정적인 차이는 무엇인가요?


Non-Repeatable Read와 Phantom Read 모두 한 트랜잭션 내에서 같은 조회를 두 번 이상 했을 때 결과가 달라지는 현상입니다. Non-Repeatable은 트랜잭션이 해당 튜플을 다시 조회할 때 다른 트랜잭션에 의해 UPDATE로 갱신되어 값이 달라지는 현상이고, Phantom Read는 트랜잭션이 조회 후 범위 UPDATE를 하려고 할 때, 다른 트랜잭션에 의해 INSERT나 DELETE로 해당 범위에 있느 튜플이 삽입, 삭제가 되어 다시 조회하였 을 때 결과가 달라지는 현상입니다.


## Q) MySQL(InnoDB)은 락(Lock)을 걸지 않고도 어떻게 동시성을 유지하며 일관된 읽기를 보장하나요? (MVCC)


InnoDB는 MVCC (다중 버전 동시 제어)라는 기술을 이용하여, 특정 트랜잭션이 해당 튜플을 UPDATE하여 갱신하면, 이전의 원본 데이터를 Undo Log에 복사해둡니다. 그리고 다른 트랜잭션이 해당 튜플에 접근하여 조회하려고 할 때 InnoDB엔진은 트랜잭션의 아이디를 비교하여 다른 트랜잭션이라고 판단되면, Undo Log에서 이전의 원본 데이터를 가져와 던져줍니다. 이를 통해 락을 걸필요가 없으므로, 성능을 극대화할 수 있습니다.


## Q) 가장 안전한 격리 수준인 SERIALIZABLE을 실무에서 사용하지 않는 이유는 무엇인가요?


SERIALIZABLE은 뮤텍스(Mutex)처럼 완벽하게 직렬화하여 상호 배제를 강제합니다. 그래서 쓰기는 커녕 읽기도 잠금을 하여 다른 트랜잭션이 읽지 못하도록합니다. 그래서 완벽한 정합성을 보장하지만 동시 처리 성능이 급격히 하락하여 병목현상을 유발하므로 실무에서는 거의 사용하지 않습니다.
