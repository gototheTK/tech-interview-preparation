
## Q) 트랜잭션의 ACID 중 원자성과 지속성, 그리고 Undo/Redo Log의 역할은?

- 원자성 (Atomicity): 트랜잭션 내의 작업이 모두 완벽히 성공하거나 모두 실패해야합니다. (All or Nothing)
- 지속성 (Durability): 커밋된 트랜잭션의 결과는 시스템 장애(정전 등)가 발생해도 영구적으로 보존되어야 합니다.
- Undo Log: 트랜잭션 실패시, 변경 이전의 ‘체크포인트’ 시점으로 롤백(되돌리기)하여 원자성을 보장합니다.
- Redo Log: 메모리 버퍼(더피 페이지)에만 커밋되고 디스크에 쓰이지 않은 상태에서 장애가 났을 때, 재실행(복구)하여 지속성을 보장합니다.

## Q) 분산 서버 환경에서 Session 대신 JWT를 사용하는 핵심적인 이유는?


상태(Stateful)방식인 Session을 사용하면, 서버의 메모리나 외부 저장소(Redis)에 장부처럼 보관해야 하므로, 스케일 아웃(서버 증설) 시 세션 동기화 문제와 네트워크 I/O 병목이 발생합니다. 그에비해 무상태(Stateless)방식인 JWT를 이용하면, 클라이언트가 토큰을 보관하며, 서버는 DB를 뒤질 필요가 없이 비밀키를 이용한 복호화 연산만으로 여부를 검증하므로 확장성이 압도적으로 좋습니다.


## Q) 커넥션 풀(HikariCP)을 사용하는 이유와, 외부 API 지연시 발생하는 연쇄 장애는?


사용이유

- DB와의 TCP/IP 연결/해제 과정이 매우 무겁고 느리기 때문에, 미리 소수의 파이프(커넥션)을 연결해 두고 빌려쓰는 방식을 취합니다. 이를 HikariCP가 제공하는 프레임워크이고, 이상적인 풀 사이즈는 (DB의 CPU 코어 수 * 2)  + 1(SSD사용 시)입니다.

연쇄 장애 시나리오

- @Transational 내부에서 외부 API 호출(예: 결제 3초 대기)이 지연되면, 해당 톰켓 스레드가 DB 커넥션을 쥔 채 대기합니다. 이 트래픽이 몰리면 커넥션 풀이 고갈되고, 결국 요청을 받은 톰캣 스레드 풀마저 모두 멈춰 서서 전체 시스템이 먹통이 됩니다.

## Q) 왜 하필 DB는 HashTable이 아닌 B-Tree 인덱스계열을 표준으로 채택했나요?

- 가장 큰 이유는 데이터베이스의 핵심 탐색 패턴인 **범위 검색(Range Scan)**과 **정렬(Order By)** 때문입니다.
- 해시 테이블은 탐색 시간 복잡도가 $O(1)$로 매우 빠르지만, 입력값이 조금만 달라도 완전히 엉뚱한 해시값을 반환하므로 데이터의 대소 관계(순서)가 파괴되어 범위 검색이 불가능합니다.
- 반면, B-Tree는 데이터가 항상 **정렬된 상태**를 유지하는 트리 구조이므로, 부등호(`<`, `>`)나 `BETWEEN`을 활용한 순차적인 범위 검색을 매우 빠른 속도로 처리할 수 있습니다.

## Q) 클러스터드(Clustered) 인덱스와 논클러스터드(Non-Clustered) 인덱스의 결정적 차이는 무엇인가요? 영어사전과 일반 전공 서적의 맨 뒤 찾아보기(색인)을 예시로 들어주세요.

- 두 인덱스의 결정적 차이는 **'인덱스의 정렬 순서와 실제 데이터의 물리적 정렬 순서가 일치하는가?'**입니다.
- **클러스터드 인덱스 (영어 사전):** 실제 데이터 자체가 인덱스 순서대로 물리적 정렬이 되어 있습니다. 물리적인 정렬은 단 한 가지 방향으로만 가능하므로, **테이블당 오직 1개**만 생성할 수 있습니다. (주로 PK에 적용)
- **논클러스터드 인덱스 (전공서적 맨 뒤 색인):** 실제 데이터의 물리적 위치는 그대로 둔 채, 별도의 공간에 인덱스(목차)와 포인터만 생성합니다. 물리적 정렬과 무관하므로 **테이블당 여러 개**를 만들 수 있습니다.

## Q) 인덱스를 무진장 많이 걸면 안되는 이유가 무엇인가요?


인덱스 역시 결국 하드디스크에 별도로 저장되어 공간을 차지하는 **물리적인 데이터(B-Tree 파일)**입니다. 따라서 인덱스가 많아지면 `SELECT` 조회는 빨라지지만, 새로운 데이터를 삽입(`INSERT`), 수정(`UPDATE`), 삭제(`DELETE`)할 때마다 테이블뿐만 아니라 **모든 인덱스 트리를 찾아가서 갱신해야 하므로 디스크 쓰기(I/O) 부하가 폭증**합니다. 특히, 데이터가 꽉 찬 B-Tree 노드에 새로운 데이터를 억지로 끼워 넣어야 할 때 노드가 두 개로 쪼개지는 **'페이지 분할(Page Split)'**이 발생하며, 이는 엄청난 시스템 부하를 유발합니다. 따라서 조회가 잦은 핵심 컬럼에만 최소한으로 인덱스를 생성해야 합니다.
